# Resume Editing Rules

이력서(resume-data.ts) 편집 시 반드시 지켜야 할 규칙.

---

## 1. 편집 전 필수 절차

- `git pull` 후 최신 파일을 `Read`로 확인한 뒤 편집 시작
- 수정 대상 섹션의 한국어/영어 양쪽을 모두 읽고 시작
- 커밋/푸시는 사용자 요청 시에만 수행

## 2. 일관성

- 동일 사실(숫자, 기간, 라벨 등)은 summary, experience, keyExperience, detail 모든 섹션에서 통일
- 한국어와 영어는 항상 동시에 반영. 한쪽만 고치지 않는다
- 확정된 라벨
  - 도메인: "AI/3D" (커머스, XR, LLM 단독 사용 금지)
  - 데드라인: "6주 데드라인을 4주로 단축"
  - Position: "Senior Backend Engineer"

## 3. 문장 품질

- 리쿠르터/엔지니어가 맥락 없이 이해 가능해야 한다
  - Bad: "3D 자동화" → Good: "3D 모델 제작 자동화"
  - Bad: "(Eng Award)" → Good: 문장에 자연스럽게 녹이기
- 구체적 근거 없는 공허한 문장은 넣지 않는다
  - 넣으려면 정량 지표를 반드시 붙인다
- 같은 단어/패턴 연속 반복 금지
  - "수상했습니다" 연속, 같은 키워드 한 문장 내 중복 등
- 첫 문장은 짧고 강하게: 성과 먼저, 설명은 뒤로

## 4. 사실 확인

- 실제 하지 않은 일은 적지 않는다 (커머스, XR 등)
- 확정 안 된 지표는 넣지 않는다 (SAM3D 생산량 등)
- 성과가 가시화되지 않은 프로젝트는 보류한다

## 5. 렌더링 규칙 (resume-template.tsx 기준)

- summary: `.join(" ")`으로 하나의 `<p>`로 렌더링됨. 문장 수와 길이 주의
- experience description: `"  "`(공백 2개)로 시작하면 `ml-8` 인덴트 sub-bullet
  - sub-bullet 하나가 너무 길면 프린트(A4, 9pt) 시 균형이 깨진다
- `**bold**`만 지원. 다른 마크다운 문법(이탤릭, 링크 등)은 렌더링되지 않는다
- `?isDetail=true` 시 STAR(detail) 뷰가 보임. detail이 있는 keyExperience만 표시
- 빈 배열(`projects: []`, `certifications: []`, `activities: []`)은 렌더링되지 않으므로 그대로 유지

## 6. 건드리지 않는 것

- `scripts/generate-docx.py` — 별도 관리, 동기화하지 않음
- `"ai-tools": []` — 빈 배열 유지
- Skills의 C# — 유지 (Unity 경험, 면접에서 구두 설명)

## 7. 수정 제안 시

- 수정안을 제시할 때는 렌더링 결과(실제 사용자가 보는 텍스트)로 보여준다
- 길이가 긴 수정안은 Before/After 비교로 제시한다
- 여러 선택지가 있으면 A/B로 제시하고 사용자가 선택하게 한다
